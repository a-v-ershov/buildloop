# Buildloop

[English](README.md) · **Русский**

Buildloop — это набор скиллов и агентов Claude Code для разработки продуктов.

Название отражает суть: каждое изменение проходит **цикл «собрать → проверить → отгрузить»**, где отдельный агент доказывает результат, прежде чем двигаться дальше. **[Перейти к установке ↓](#установка)**

**Доведите сырую идею до готового релиза с помощью Buildloop — шаг за шагом, и каждый шаг
проходит проверку, чтобы ИИ создавал *именно то, что нужно*, и доказывал, что оно работает, а не
просто заявлял об этом.**

Три конвейера с проверками на каждом этапе — **спецификация → сборка → релиз** — устанавливаются как
плагин в любой проект прямо с GitHub. Каждый скилл отвечает на том языке, на котором вы к нему
обращаетесь.

---

## 3 проблемы, которые он решает

1. **ИИ быстро делает не то** — он перескакивает с однострочного запроса прямо к коду, и вы выкатываете отполированную версию непроверенной идеи, а понимаете это слишком поздно.
2. **ИИ сам проверяет свою работу** — один и тот же агент пишет функцию, заявляет *«работает»* и пишет тесты, которые проходят лишь потому, что он подогнал их под нужный результат.
3. **Качество и решения теряются бесследно** — дыры в безопасности, узкие места по скорости и причины принятых решений не видны в зелёном модульном тесте и пропадают между сессиями.

## Что вы получаете

1. **Три конвейера, по одной команде на каждый** — вызовите `create-project-spec`, `build-product` или `release-product`, и один мета-скилл сам проведёт вас через все фазы.
2. **Сначала он вас расспрашивает — и только потом берётся за дело** — дотошный опрос вытягивает из вас максимум контекста, вплоть до предпочтений по стеку и стилю кода, по одному вопросу за раз и всегда с рекомендованным ответом.
3. **Работу всегда проверяет отдельный агент** — ничто не подтверждает само себя: рецензент на каждой фазе спецификации плюс новый, независимый проверяющий, который пишет придирчивые тесты, а хук запрещает ему трогать код.
4. **Настоящий цикл разработки + релизные аудиты, выискивающие ошибки самого ИИ** — обязательный контроль качества локально, затем независимые аудиты безопасности / производительности / качества / доступности / документации перед выпуском.

**Плюс:** отвечает на вашем языке · восстанавливает спецификацию из готового кода (brownfield) ·
оставляет в репозитории зафиксированную память проекта — документы, ADR, бэклог, карту в `CLAUDE.md` —
которая не устаревает · поставляется как обычный плагин Claude Code, без отдельной среды выполнения и
MCP-сервера.

## Как это работает

Три конвейера, каждый — тонкий **оркестратор**, который дирижирует узкоспециализированными
под-скиллами; каждый под-скилл работает и сам по себе.

### 1 — Спецификация: идея → готовая к сборке спецификация

`create-project-spec` проходит семь фаз, у каждой своя роль; каждая исследует свои утверждения (со
ссылками на источники), пишет черновик, проходит придирчивый разбор у рецензента и выдаёт
исследовательский документ и короткое резюме для человека в `.buildloop/project-spec/`.

| Шаг | Скилл | Роль | Результат |
|------|-------|------|------------|
| 1 | `gather-context` | Интервьюер-исследователь | `project-brief` — расспрашивает вас, пока у вас не сложится общее понимание |
| 2 | `validate-idea` | Основатель-инвестор | `idea-validation` — KILL / SHRINK / вопросы, заставляющие определиться |
| 3 | `define-product-requirements` | Менеджер продукта | `product-requirements` — полный набор функций, каждая с проверяемыми критериями |
| 4 | `create-user-flows` | Продуктовый дизайнер | `user-flows` — как пользователи проходят по продукту |
| 5 | `define-design-decisions` | Ведущий по дизайн-системе | `design-decisions` — направление, а не макеты; переход к технической части |
| 6 | `design-architecture` | Архитектор ПО | `architecture` + ADR — сначала сценарии качества, затем компоненты + технологии |
| 7 | `design-dev-architecture` | Инженер DX / платформы | `dev-architecture` + ADR — локальный внутренний цикл и ИИ-инструментарий |

> **Brownfield (готовый код):** направьте его на существующий код (`project_type: existing`), и первым
> запустится `map-codebase`, который зафиксирует фактическое состояние; затем каждая фаза реконструирует
> *целевую* спецификацию и фиксирует расхождения.

### 2 — Сборка: спецификация → работающее ПО

`build-product` превращает спецификацию в код последовательно — по одной задаче за раз, в едином
рабочем дереве, без параллелизма — изменяя настоящий репозиторий.

| Шаг | Скилл | Роль |
|------|-------|------|
| 1 | `setup-dev-environment` | Разворачивает репозиторий, поднимает весь стек одной командой, выстраивает обязательный контроль качества (`make check` + хуки) |
| — | `create-design-system` | *(UI)* превращает направление дизайна в конкретику — зафиксированный в репозитории `DESIGN.md`, который вы выбираете из отрисованных вариантов |
| 2 | `plan-development` | Формирует канбан-бэклог — по файлу на задачу, где `blocked_by` *и есть* граф зависимостей |
| 3 | `implement-feature` | Реализует одну задачу и добивается, чтобы контроль качества был пройден |
| 4 | `verify-feature` | **Новый, отдельный агент**, который пишет придирчивые тесты, гоняет настоящий стек и доказывает наблюдаемый результат |

Он берёт готовую к работе задачу с наименьшим id, реализует её, проверяет отдельным агентом (с
ограничением по числу попыток — на пределе задача помечается как `needs_human`) и, когда контроль
качества пройден, делает контрольный коммит. `generate-mockups` по запросу отрисовывает варианты
интерфейса; `propagate-changes` переносит более позднюю правку спецификации дальше — в бэклог.

### 3 — Релиз: работающее ПО → готовый релиз

`release-product` доказывает сквозные свойства, которые не под силу ни одной отдельной задаче, и затем
выпускает; аудиты работают только на чтение, поэтому запускаются **параллельно** и оформляют находки
как задачи на доработку, а не правят код сами.

| Скилл | Что доказывает |
|-------|----------------|
| `audit-security` | модель угроз STRIDE-lite — секреты, авторизация, инъекции, «смертельная троица» |
| `audit-performance` | сценарии атрибутов качества — измеренные p95, пропускная способность, N+1, стоимость |
| `audit-product` | пользовательские сценарии, из конца в конец и в связке между функциями |
| `audit-code-health` | дублирование, мёртвый код, долг из подавленных предупреждений, качество тестов (мутационное тестирование) |
| `audit-accessibility` | цель по WCAG — клавиатура, фокус, программа чтения с экрана, контраст |
| `cut-release` | чистое дерево + нет открытых 🔴 → документы, версия, журнал изменений, тег, коммит, PR (всегда с подтверждением; **останавливается перед развёртыванием в продакшене**) |

Блокирующая находка становится задачей на доработку, исправляется запуском `build-product`, а затем
**повторно проверяется аудитом, чтобы подтвердить, что она закрыта**.

## Сравнение

Большинство инструментов закрывают лишь один отрезок пути; Buildloop делает ставку на весь путь
целиком — плюс отдельный оппонент, проверяющий каждый шаг.

| Инструмент | Что это | Чем отличается Buildloop |
|------|---------|-------------------------------|
| **[GitHub Spec Kit](https://github.com/github/spec-kit)** | Spec → Plan → Tasks → Implement; шаблоны, не привязан к агенту | Останавливается на реализации — нет независимой проверки, релизных аудитов, встроенного исследования и придирчивого разбора. |
| **[BMAD-METHOD](https://github.com/bmad-code-org/BMAD-METHOD)** | Agile-фреймворк на агентах, 12+ специализированных агентов | Планирует и собирает, но нет фазы релизных аудитов только на чтение, нет проверяющего, встроенного в сам инструмент, нет реконструкции из готового кода. |
| **[Task Master](https://github.com/eyaltoledano/claude-task-master)** | PRD → разбиение на задачи; MCP-менеджер проекта | Только управление задачами — не проверяет идею, не исследует, не проектирует и не проводит аудит; требует запущенного MCP-сервера. |
| **[gstack](https://github.com/garrytan/gstack)** | 23 ролевых скилла для цикла «на каждый PR» | Хорош в цикле разбора/выпуска на готовом коде, но без генератора «идея → спецификация» и без реконструкции спецификации. *(Наш ориентир по дизайну.)* |
| **[SuperClaude](https://github.com/SuperClaude-Org/SuperClaude_Framework)** | ~30 команд + ролевые агенты, внедряемые в `CLAUDE.md` | Набор команд «по меню», а не последовательный конвейер с проверками на каждом этапе, куда встроены исследование, разбор и разделение «автор/рецензент». |

## Установка

В любом проекте, внутри Claude Code:

```
/plugin marketplace add https://github.com/a-v-ershov/buildloop.git
/plugin install buildloop@buildloop
```

Скиллы появляются с префиксом `buildloop:<skill>`. Начните проект с
**`/buildloop:create-project-spec`** и ответьте на три вопроса настройки — или запускайте любой
отдельный скилл по отдельности.

## Обновление

Установка подхватывает изменения только после **повышения** версии плагина (`version`). Чтобы обновить
проект, где он уже установлен (обновление применяется после перезапуска):

```
/plugin marketplace update buildloop
/plugin update buildloop@buildloop
```

Или включите автообновление один раз: `/plugin` → **Marketplaces** → `buildloop` →
**Enable auto-update**.

## Справочник скиллов

Все 25 скиллов, сгруппированные по конвейерам. Оркестраторы идут первыми в каждой группе; каждый
под-скилл работает и сам по себе.

### Спецификация — идея → готовая к сборке спецификация

| Скилл | Роль | Что делает | Пишет |
|-------|------|------------|-------|
| `create-project-spec` | Оркестратор | Выстраивает семь фаз спецификации от сырой идеи до готовой к сборке | спецификацию |
| `map-codebase` | Археолог кода | *(brownfield)* Реконструирует фактическое состояние готовой кодовой базы | `codebase-map.research.md` |
| `gather-context` | Интервьюер-исследователь | Расспрашивает вас, превращая короткий бриф в общее понимание | `project-brief.research.md` |
| `validate-idea` | Основатель-инвестор | Проверяет на прочность спрос, аудиторию, проблему и бизнес-модель | `idea-validation.research.md` |
| `define-product-requirements` | Менеджер продукта | Определяет, для кого продукт, и весь зафиксированный набор функций вместе с критериями | `product-requirements.research.md` |
| `create-user-flows` | Продуктовый дизайнер | Описывает, как пользователи проходят путь к ценности | `user-flows.research.md` |
| `define-design-decisions` | Ведущий по дизайн-системе | Задаёт направление дизайна — система, ключевые экраны, платформы, доступность | `design-decisions.research.md` |
| `design-architecture` | Архитектор ПО | Сначала сценарии качества, затем компоненты + реализующие их технологии | `architecture.research.md` + `adr/` |
| `design-dev-architecture` | Инженер DX / платформы | Проектирует локальный внутренний цикл, тестирование под управлением ИИ и ИИ-инструментарий | `dev-architecture.research.md` + `adr/` |

### Сборка — спецификация → работающее ПО

| Скилл | Роль | Что делает | Пишет |
|-------|------|------------|-------|
| `build-product` | Оркестратор | Берёт по одной готовой к работе задаче за раз и ведёт цикл реализация → проверка → коммит | цикл сборки |
| `setup-dev-environment` | Инженер платформы | Разворачивает репозиторий, поднимает стек, настраивает контроль качества | репозиторий + `.buildloop/project-setup/` |
| `create-design-system` | Ведущий по дизайн-системе | *(UI)* Превращает направление дизайна в конкретику из отрисованных вариантов | `DESIGN.md` + `design-system.md` |
| `plan-development` | Ведущий разработки | Превращает спецификацию в канбан-бэклог, по файлу на задачу | `.buildloop/build-plan/` |
| `implement-feature` | Разработчик | Реализует одну задачу из бэклога и добивается, чтобы контроль качества был пройден | код |
| `verify-feature` | Независимый проверяющий | Пишет придирчивые тесты и доказывает наблюдаемые результаты готовой задачи | тесты + вердикт |
| `generate-mockups` | Прототипировщик интерфейса | *(по запросу)* Отрисовывает черновые варианты интерфейса по `DESIGN.md` для сравнения | одноразовые макеты |

### Релиз — работающее ПО → готовый релиз

| Скилл | Роль | Что делает | Пишет |
|-------|------|------------|-------|
| `release-product` | Капитан релиза | Запускает аудиты, заводит задачи на доработку, ведёт исправления, перепроверяет и затем выпускает | цикл релиза |
| `audit-security` | Инженер по безопасности | Доказывает модель угроз STRIDE-lite на работающей системе | `security-audit.md` |
| `audit-performance` | Инженер по производительности | Сопоставляет показатели системы со сценариями атрибутов качества | `performance-audit.md` |
| `audit-product` | Ведущий QA | Проводит пользовательские сценарии из конца в конец, в том числе в связке между функциями | `qa-report.md` |
| `audit-code-health` | Старший инженер | Измеряет деградацию кода, долг из подавлений и качество набора тестов | `code-health-audit.md` |
| `audit-accessibility` | Специалист по доступности | *(UI)* Проверяет интерфейс на соответствие цели по WCAG | `accessibility-audit.md` |
| `cut-release` | Релиз-инженер | Поднимает версию, обновляет журнал изменений, ставит тег, делает коммит и PR — с проверками, останавливается перед развёртыванием | релизные документы + PR |

### Сквозные — используются по всем конвейерам

| Скилл | Роль | Что делает | Пишет |
|-------|------|------------|-------|
| `commit` | Помощник по git | Разбивает изменения сессии на аккуратные коммиты (сообщения на английском) | коммиты |
| `propagate-changes` | Сквозной дирижёр | Согласует нижестоящие документы и бэклог после правки вышестоящей спецификации | обновлённые документы + бэклог |

К каждому `*.research.md` прилагается парный `*.summary.md`; документы спецификации лежат в
`.buildloop/project-spec/`, аудиты — в `.buildloop/release/`. *(brownfield)* = только проекты на готовом коде,
*(UI)* = только проекты с интерфейсом (иначе скилл сам себя пропускает), *(по запросу)* = не
запускается автоматически в конвейере. `gather-context` можно вызвать и отдельно — как точечный опрос.
